# Review Feedback Apply Skill

## 목표

- `reviews/*.md` 피드백을 읽고 챕터 콘텐츠 파일에 반영한다.
- 반영 및 확인이 끝난 리뷰 파일만 삭제한다.
- 미해결 항목이 있으면 파일을 삭제하지 않고 남긴다.

## 실행 절차

1. 처리할 피드백 파일 수집.

```bash
rg --files reviews -g '*.md' | sort
```

파일이 없으면 "반영할 리뷰 없음"을 보고하고 종료한다.

2. 각 리뷰 파일을 대상 챕터 콘텐츠 파일에 매핑.

- `chapter-01-review.md` 같은 파일명에서 챕터 번호를 파싱한다.
- `src/content/chapters/<chapter-number>-*.mdx` 패턴으로 대상 파일을 찾는다.
- 매칭이 0개 또는 여러 개면 리뷰 본문의 챕터 제목/URL 단서를 읽어 재매핑한다.
- 그래도 모호하면 수정 전에 사용자에게 확인한다.

3. 피드백에서 실행 가능한 수정 항목 추출.

- 명시적 수정 요청과 높은 우선순위 항목부터 처리한다.
- 필수 수정과 선택 제안을 구분한다.

4. 콘텐츠 수정 반영.

- 프로젝트의 문체와 용어를 유지한다.
- 요청된 경우 개념 순서, 용어 첫 정의 누락, 갑작스러운 전문용어 등장, 섹션 제목-내용 불일치를 수정한다.
- 피드백과 무관한 파일은 수정하지 않는다.

5. 삭제 전 검증.

- 수정한 섹션을 다시 읽고 각 수정 항목의 반영 여부를 확인한다.
- 가능하면 `pnpm run lint` 또는 범위가 더 작은 검증 명령을 실행한다.
- 검증 실패 또는 미반영 항목이 있으면 해당 리뷰 파일은 삭제하지 않는다.

6. 완료된 리뷰 파일만 삭제.

- 반영 + 검증이 끝난 리뷰 markdown 파일만 삭제한다.
- 삭제 후 `reviews/*.md`를 다시 조회해 남은 파일을 확인한다.

7. 완료 보고.

- 수정한 콘텐츠 파일 목록을 보고한다.
- 반영한 핵심 피드백을 요약한다.
- 삭제한 리뷰 파일과 남겨둔 리뷰 파일(사유 포함)을 함께 보고한다.

## 삭제 규칙

아래 조건을 모두 만족할 때만 리뷰 파일을 삭제한다.

- 관련 콘텐츠 수정이 완료됨.
- 최소 1회 검증을 수행함.
- 해당 리뷰 파일에 blocker가 남아있지 않음.

조건을 하나라도 만족하지 못하면 `reviews/`에 파일을 남기고 사유를 보고한다.
