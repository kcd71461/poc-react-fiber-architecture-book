# 01. 왜 Fiber가 필요했나

## 비유 소개

작은 카페에 계산대가 하나뿐이면, 주문이 복잡한 손님 한 명 때문에 뒤에 선 간단 주문까지 모두 기다리게 됩니다.
React 15 시절 렌더링 모델도 비슷했습니다.

그 시절 React에서 트리를 조정하던 엔진을
<InlineAnnotation term="Stack Reconciler">
  React 15까지의 재귀 기반 Reconciler 구현입니다. 한 번 렌더링이 시작되면 중간에
  끊고 긴급 작업을 끼워 넣기 어려워, 긴 트리에서는 입력 지연이 커질 수 있었습니다.
</InlineAnnotation>
라고 부릅니다.

- 렌더링이 시작되면 재귀 호출이 길게 이어짐
- 사용자 입력보다 렌더링 계산이 먼저 CPU를 점유할 수 있음
- 브라우저 입장에서는 "입력은 왔는데 반응이 늦는" 상태가 됨

아래 시뮬레이션은 한{" "}
<InlineAnnotation term="frame budget">
  브라우저가 한 프레임(예: 60Hz에서 약 16.7ms) 안에 JS 실행, 레이아웃, 페인트를
  마쳐야 하는 시간 한도입니다. 이 한도를 반복적으로 넘기면 끊김(jank)이나 입력
  지연이 눈에 띄게 늘어납니다.
</InlineAnnotation>
(16.7ms) 안에서 Stack 방식과 Fiber 방식이 시간을 쓰는 차이를 단순화해 보여줍니다.

<BlockingVsYieldingTimeline frameBudgetMs={16.7} stackTaskMs={38} fiberSliceMs={5} />

## 문제 정의

먼저 용어보다 현상부터 보면 이해가 쉽습니다.

- 입력했는데 커서/화면 반영이 늦다
- 스크롤 중에 미세한 끊김이 반복된다
- 무거운 렌더 구간에서 UI가 잠깐 멈춘 것처럼 보인다

이런 상태를 React 맥락에서
<InlineAnnotation term="Stop-the-world">
  GC 교과서 용어를 그대로 가져온 엄밀한 정의가 아니라, 렌더 계산이 길어져 메인
  스레드를 점유하면서 사용자 체감 상 "세상이 잠깐 멈춘 것처럼" 보이는 상황을
  가리키는 표현입니다.
</InlineAnnotation>
성격의 문제라고 부릅니다.

문제를 정리하면 다음과 같습니다.

1. 재귀 기반 순회는 "지금 당장 멈추고 다른 일 처리"가 어려움
2. 컴포넌트 트리가 깊거나 계산량이 크면 frame budget을 초과하기 쉬움
3. 긴급 업데이트(입력 반영)와 비긴급 업데이트가 같은 경로에서 경합함

## 해결 방법

React 16에서 Fiber는 "트리를 단번에 끝내는 방식" 대신
**작업 단위(unit of work)를 쪼개고, 필요 시 브라우저에 제어권을 양보(yield)하는 방식**으로 전환했습니다.

핵심 아이디어는 세 축입니다.

1. <InlineAnnotation term="Cooperative Multitasking">
     cooperative multitasking은 긴 작업을 작은 조각으로 나눠, 조각 사이마다 제어권을
     돌려줄 수 있게 만드는 실행 전략입니다.
   </InlineAnnotation>
   Fiber는 이 관점을 렌더링 루프에 적용해 긴 작업을 분할합니다.
2. <InlineAnnotation term="Algebraic Effects (심화)">
     책갈피처럼 "멈춘 위치"를 기억했다가 다시 이어 실행할 수 있다는 사고방식입니다.
     Fiber는 이 아이디어를 참고해 중단/재개 가능한 실행 모델을 설계했습니다.
   </InlineAnnotation>
   관점에서 멈춘 위치를 다시 이어갈 수 있는 중단/재개 모델을 설명할 수 있습니다.
3. 우선순위 모델: 사용자 입력 같은 긴급 작업을 먼저 처리해 체감 응답성을 지킵니다.

## 배경 노트

> React 16 Fiber 전환은 Sebastian Markbage와 Andrew Clark 등이 주도했습니다.
> 설계 아이디어는 OS의 **cooperative multitasking**과 **algebraic effects**에서 영감을 받았습니다.

## 기술적 구현 (TypeScript 의사 코드)

```ts
// Stack Reconciler 스타일: 끝날 때까지 재귀적으로 진행
function renderStack(node: VNode): void {
  for (const child of node.children) {
    renderStack(child);
  }
  commit(node);
}

// workLoop는 nextUnitOfWork를 시작점으로 삼아 순회를 시작한다.
// nextUnitOfWork는 최초에 루트 Fiber에서 시작한다.
// workLoop는 매 반복마다 이 포인터를 갱신하면서 중단/재개 지점을 기록한다.
let nextUnitOfWork: FiberNode | null = rootFiber;

// Fiber 스타일: 시간 예산을 넘기면 중단하고 다음 틱에 재개
function workLoop(deadline: number): void {
  while (nextUnitOfWork !== null && performance.now() < deadline) {
    nextUnitOfWork = performUnitOfWork(nextUnitOfWork);
  }

  if (nextUnitOfWork !== null) {
    scheduleHostCallback();
    return;
  }

  commitRoot();
}
```

workLoop는 위에서 초기화한 `nextUnitOfWork`를 기준으로 실행을 이어가고,
아래 `performUnitOfWork`가 실제로 "다음 작업 단위"를 계산합니다.

Fiber 노드는 `child`, `sibling`, `return` 포인터로 트리를 연결합니다.
아래 코드는 그 포인터를 따라 순회 패턴을 구현한 핵심 로직입니다.

```ts
// 핵심은 "상태 저장 후 재개"다.
function performUnitOfWork(fiber: FiberNode): FiberNode | null {
  beginWork(fiber);

  // 1) 가능하면 자식으로 먼저 내려간다.
  if (fiber.child !== null) {
    return fiber.child;
  }

  // 2) 자식이 없으면 완료 단계로 올라가며 다음 형제를 찾는다.
  let node: FiberNode | null = fiber;
  while (node !== null) {
    completeWork(node);

    // 2-a) 형제가 있으면 그쪽으로 이동
    if (node.sibling !== null) {
      return node.sibling;
    }

    // 2-b) 형제가 없으면 부모로 복귀
    node = node.return;
  }

  return null;
}
```

## 실습 예제 (코드리뷰형)

### 시나리오

검색창 입력 시 화면이 밀리고, DevTools에서 특정 업데이트가 30~50ms씩 길게 걸립니다.

```tsx
function SearchPage({ items }: { items: Item[] }) {
  const [query, setQuery] = useState("");

  // heavyScore: 고의로 느리게 만든 점수 계산 함수(렌더 병목 시뮬레이션 목적)
  const filtered = items
    .filter((item) => item.name.includes(query))
    .sort((a, b) => heavyScore(b) - heavyScore(a));

  return (
    <>
      <input value={query} onChange={(e) => setQuery(e.target.value)} />
      <ResultList items={filtered} />
    </>
  );
}
```

여기서 `heavyScore`는 실제 서비스의 복잡한 계산을 흉내 내기 위해 의도적으로 느리게 만든 함수입니다.

## 오해 바로잡기

1. **"Fiber면 무조건 빨라진다"**
   - 아니다. Fiber는 평균 응답성을 개선하는 실행 모델이고, 총 연산량을 자동으로 줄여주진 않는다.

2. **"Render가 길면 Commit도 길다"**
   - 반드시 그렇지 않다. Render 병목과 Commit 병목은 원인과 해결책이 다르다.
   - Commit Phase 구조와 병목 포인트는 03장에서 상세히 다룬다.

3. **"이건 Concurrent Mode 쓸 때만 해당"**
   - 아니다. Fiber 구조 이해는 React 18 실무 최적화 전반(`memo`, `transition`, profiler 해석)에 직접 연결된다.

## 요약 체크리스트
