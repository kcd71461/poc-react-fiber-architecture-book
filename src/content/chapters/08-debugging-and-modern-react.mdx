# 08. 디버깅과 현대 React

지금까지 7개 장에 걸쳐 Fiber의 내부 설계를 살펴보았습니다.
이 마지막 장에서는 그 지식을 실무 디버깅 도구와 최신 React 기능에 연결합니다.

## 비유 소개

자동차 계기판은 엔진 내부를 몰라도 읽을 수 있지만,
정비 매뉴얼을 함께 알면 "RPM이 왜 높은지", "온도 경고가 어떤 부품 문제인지" 정확히 진단할 수 있습니다.

React DevTools Profiler도 마찬가지입니다.
Fiber 내부를 모르면 그래프를 보기만 하지만,
Fiber를 이해하면 그래프가 **진단 도구**가 됩니다.

## 문제 정의

실무에서 성능 문제를 다룰 때 흔한 어려움:

- Profiler의 Flame Graph에서 "어떤 컴포넌트가 오래 걸렸는지"는 보이지만, **왜** 오래 걸렸는지 모른다
- Render와 Commit 중 어디가 병목인지 구분하지 못한다
- Server Components, Streaming SSR 같은 최신 기능이 기존 Fiber 모델과 어떻게 연결되는지 모른다

## 해결 방법

이 장에서는 두 가지를 다룹니다.

1. **Profiler 해석**: Fiber 트리 순회, Lane, Render/Commit 분리 지식을 Profiler 데이터에 적용
2. **현대 React 기능**: Server Components와 Streaming SSR이 Fiber 아키텍처 위에서 어떻게 동작하는지 이해

## React DevTools Profiler: Fiber 관점으로 읽기

### Flame Graph와 Fiber 트리

<InlineAnnotation term="Flame Graph">
  React DevTools Profiler에서 각 컴포넌트의 렌더링 시간을 가로 막대로 시각화한 차트입니다.
  막대의 너비가 렌더링 소요 시간을 나타내며, 상하 관계가 Fiber 트리의 부모-자식 관계에 대응합니다.
</InlineAnnotation>
는 Fiber 트리 순회의 시각적 표현입니다.

```
Profiler Flame Graph:
┌────────────────────────── App (12ms) ──────────────────────────┐
│ ┌─── Header (0.5ms) ──┐ ┌──────── Main (10ms) ────────┐      │
│ │                      │ │ ┌── List (8ms) ─┐ ┌Side(1ms)│      │
│ │                      │ │ │ Item×100      │ │         │      │
│ └──────────────────────┘ └─┴───────────────┴─┴─────────┘      │
└───────────────────────────────────────────────────────────────────┘

↕ 대응하는 Fiber 트리:
  App
  ├── Header (child)
  └── Main (sibling of Header)
      ├── List (child)
      │   └── Item×100
      └── Sidebar (sibling of List)
```

Fiber 트리에서 Header는 App.child이고, Main은 Header.sibling입니다 (03장 참조).
Flame Graph에서는 이 sibling 관계가 같은 수준의 가로 막대로 나타납니다.

Flame Graph를 읽을 때 Fiber 관점에서 주목할 점:

| Profiler 신호 | Fiber 해석 | 대응 전략 |
| --- | --- | --- |
| 특정 막대가 비정상적으로 넓음 | 해당 Fiber의 beginWork/completeWork가 오래 걸림 | 컴포넌트 내부 계산 최적화 |
| 같은 컴포넌트가 매 렌더마다 등장 | bailout이 발생하지 않음 (lanes에 매번 포함) | React.memo, useMemo로 bailout 조건 추가 |
| 자식 컴포넌트 전체가 함께 넓어짐 | childLanes로 인해 서브트리 전체가 렌더 대상 | 상태를 가능한 아래로 내려 렌더 범위 축소 |
| Commit 시간이 Render보다 길음 | DOM mutation이 많거나 useLayoutEffect가 무거움 | 변경량 축소, effect 재배치 |

### Render vs Commit 분리 진단

Profiler의 "Ranked" 탭에서 각 컴포넌트의 렌더링 시간을 확인하면 Render Phase 비용을 볼 수 있고,
"Commit" 정보에서는 실제 DOM 반영 비용을 확인할 수 있습니다.

```
진단 플로우차트:

성능 문제 감지
  │
  ├─ Render 시간이 길다?
  │   ├─ 특정 컴포넌트가 원인? → React.memo, useMemo, 컴포넌트 분할
  │   └─ 전체적으로 느리다? → useTransition으로 Lane 분리, 가상화
  │
  └─ Commit 시간이 길다?
      ├─ DOM mutation이 많다? → 변경 범위 축소, key 최적화 (05장)
      └─ useLayoutEffect가 무겁다? → useEffect로 이동 가능한지 검토
```

### 실무 디버깅 체크리스트

성능 문제를 마주했을 때 Fiber 지식을 활용하는 단계별 체크리스트:

1. **Profiler로 Flame Graph 캡처**
   - 어떤 컴포넌트가 렌더 대상인지 확인 (Fiber 트리에서 lanes가 설정된 노드)
   - 불필요하게 렌더되는 컴포넌트가 있는지 확인

2. **Render 병목 분석**
   - 해당 컴포넌트의 props/state가 실제로 변경되었는지 확인
   - bailout 조건(React.memo, shouldComponentUpdate)이 누락되었는지 확인
   - 계산이 무거운 부분은 useMemo로 캐싱 가능한지 검토

3. **Commit 병목 분석**
   - DOM 변경량이 과도한지 확인 (key 전략 검토, 05장)
   - useLayoutEffect에서 무거운 DOM 측정/조작이 있는지 확인
   - 필요하면 useEffect로 이동 (비동기 처리)

4. **Lane 분리 적용**
   - 긴급/비긴급 업데이트가 같은 Lane에서 경합하는지 확인
   - useTransition이나 useDeferredValue 적용 검토 (07장)

## Server Components와 Fiber

### Server Components의 배경

<InlineAnnotation term="Server Components (RSC)">
  서버에서 렌더링되어 직렬화된 결과(React Server Component Payload)로 클라이언트에
  전달되는 컴포넌트입니다. 클라이언트에서 JavaScript 번들에 포함되지 않아
  번들 크기를 줄이고 초기 로딩을 개선합니다.
</InlineAnnotation>
는 Fiber 아키텍처의 확장입니다.

기존 Fiber 모델에서는 모든 컴포넌트가 클라이언트에서 렌더링됩니다.
Server Components는 이 경계를 확장해, 일부 컴포넌트를 서버에서 미리 처리합니다.

### 동작 흐름

```
서버 측:
  ServerComponent → 렌더링 → RSC Payload (직렬화된 Fiber 정보)
                                    │
                              네트워크 전송
                                    │
클라이언트 측:                       ▼
  RSC Payload 수신 → Fiber 트리에 통합 → Client Components만 hydration
```

### Fiber 관점에서의 통합

아래는 Server Components의 Fiber 통합 과정을 단순화한 의사 코드입니다.
실제 React에서는 `react-client` 패키지의 Flight 모듈이 이 역할을 수행합니다.

```ts
// Server Component의 Fiber 처리 (개념적 의사 코드)
function resolveServerComponent(
  fiber: FiberNode,
  serverPayload: ServerComponentPayload
): FiberNode {
  // 서버에서 이미 렌더링된 결과를 Fiber 노드로 변환
  const serverFiber = createFiberFromServerPayload(serverPayload);

  // 클라이언트 Fiber 트리에 삽입
  serverFiber.return = fiber;

  // Server Component는 클라이언트에서 재렌더링되지 않음
  // → lanes가 설정되지 않으므로 Reconciliation에서 bailout
  serverFiber.lanes = NoLane;

  return serverFiber;
}
```

Server Components가 Fiber 트리에 통합되는 핵심:

- 서버에서 렌더링된 결과가 RSC Payload로 직렬화되어 전달된다
- 클라이언트는 이 payload를 Fiber 노드로 변환해 트리에 삽입한다
- Server Component Fiber는 lanes가 설정되지 않으므로, 클라이언트에서 재렌더링되지 않는다
- Client Component만 hydration 과정을 거친다

    서버에서 미리 처리된 Fiber 노드가 클라이언트 트리에 통합되는 구조입니다.

## Streaming SSR과 Suspense

### 기존 SSR의 한계

전통적인 SSR에서는 전체 페이지를 서버에서 렌더링한 뒤 한 번에 전송합니다.
느린 데이터 소스가 하나라도 있으면 전체 페이지 전송이 블로킹됩니다.

### Streaming SSR의 해결

<InlineAnnotation term="Streaming SSR">
  서버에서 HTML을 한 번에 보내는 대신, 준비된 부분부터 점진적으로 스트리밍하는 방식입니다.
  Suspense boundary를 기준으로 독립적인 청크로 나뉘어 전송됩니다.
</InlineAnnotation>
은 Suspense와 결합해 이 문제를 해결합니다.

```
전통적 SSR:
  서버: [Header + Main + Sidebar 전체 렌더링] ──────────→ 클라이언트
  (전체 완료까지 대기)

Streaming SSR (Suspense boundary가 청크를 결정):
  서버: [Suspense-1: Header 즉시 전송]  ─→ 클라이언트: Header 표시
        [Suspense-2: Main 준비됨]       ──→ 클라이언트: Main 삽입
        [Suspense-3: Sidebar 준비됨]    ─→ 클라이언트: Sidebar 삽입
```

### Suspense가 Streaming의 경계를 정의

```tsx
// 이 구조가 Streaming SSR의 청크 경계를 결정
<Suspense fallback={<HeaderSkeleton />}>
  <Header />  {/* 즉시 렌더 가능 → 첫 번째 청크 */}
</Suspense>

<Suspense fallback={<ContentSpinner />}>
  <MainContent />  {/* 데이터 필요 → 데이터 준비되면 청크 전송 */}
</Suspense>

<Suspense fallback={<SidebarSkeleton />}>
  <Sidebar />  {/* 별도 데이터 → 독립 청크 */}
</Suspense>
```

각 Suspense boundary가 독립적인 스트리밍 단위가 됩니다.

### Hydration과 Fiber

<InlineAnnotation term="Selective Hydration">
  서버에서 전송된 HTML에 이벤트 핸들러를 연결하는 과정(hydration)을
  Suspense boundary 단위로 분할해 점진적으로 수행하는 방식입니다.
  사용자가 상호작용하는 영역을 우선 hydration해 체감 응답성을 높입니다.
</InlineAnnotation>
은 Fiber의 우선순위 시스템을 hydration에 적용한 것입니다.

서버에서 받은 HTML은 정적 마크업이므로 이벤트 핸들러가 아직 연결되지 않은 상태입니다.
이 상태에서 인터랙션을 가능하게 만드는 과정이 hydration입니다.

```
Selective Hydration 흐름:

서버 HTML 수신 → 화면 표시 (JS 미로드, 인터랙션 불가)
                    │
  ┌─────────────────┼─────────────────┐
  ▼                 ▼                 ▼
Header hydration  Main hydration   Sidebar hydration
(우선순위: 높음)   (우선순위: 보통)  (우선순위: 보통)

사용자가 Sidebar를 클릭하면?
→ Sidebar의 hydration 우선순위가 높아짐 (Lane 승격)
→ Main보다 Sidebar를 먼저 hydration
```

핵심:
- React 내부에서 hydration은 별도의 Hydration Lane을 사용한다
- 사용자 상호작용이 감지되면, 해당 Suspense boundary의 hydration이 더 높은 우선순위 Lane으로 재스케줄링된다
- 이는 06장의 `markStarvedLanesAsExpired`와는 다른 메커니즘이지만, "우선순위를 동적으로 조정한다"는 같은 설계 철학을 따른다
- Fiber의 우선순위 시스템이 렌더링뿐 아니라 hydration까지 확장된 구조다

## 외부 라이브러리와 React 스케줄링

실무에서는 차트 라이브러리(D3, Chart.js), 에디터(Monaco, CodeMirror), 지도(Mapbox) 등
외부 라이브러리가 React 바깥에서 직접 DOM을 조작하는 경우가 많습니다.
이때 React의 Fiber 스케줄링과 충돌이 발생할 수 있습니다.

**주요 충돌 패턴:**
- 외부 라이브러리가 `ref.current`를 통해 Commit Phase 완료 전에 DOM에 접근
- 라이브러리의 자체 렌더링 루프와 React의 Render/Commit 타이밍 불일치
- `useTransition`으로 낮춘 우선순위 업데이트가 외부 라이브러리의 동기 업데이트와 불일치

**대응 전략:**
1. **`useLayoutEffect`** 사용: Commit 직후, 브라우저 페인트 전에 외부 라이브러리를 동기화
2. **`flushSync`** 활용: 외부 라이브러리와 동기화가 필요한 시점에 React 업데이트를 즉시 반영
3. **Suspense boundary 분리**: 외부 라이브러리 영역을 별도 boundary로 격리해 React 렌더링과 독립적으로 관리

## 실습 예제 (코드리뷰형)

### 실습 A: Profiler 기반 디버깅

#### 시나리오

전자상거래 페이지에서 상품 목록 로딩 시 전체 페이지가 빈 상태로 오래 머무는 문제가 있습니다.
Profiler를 열어보니 Render Phase 30ms, Commit Phase 15ms로 둘 다 높습니다.

```tsx
function ProductPage() {
  const products = useFetchProducts(); // useEffect + useState 패턴
  const reviews = useFetchReviews();   // useEffect + useState 패턴

  return (
    <div>
      <ProductList items={products} />
      <ReviewSection reviews={reviews} />
    </div>
  );
}
```

#### 질문

1. Profiler에서 Render 30ms / Commit 15ms를 각각 어떻게 해석하는가?
2. Render 병목과 Commit 병목의 해결 방향이 어떻게 다른가?

#### 답안 체크리스트

- [ ] Render 병목(30ms): 데이터 요청 대기와 컴포넌트 렌더가 같은 Phase에서 처리되는 문제를 지적했다.
- [ ] Commit 병목(15ms): 대량의 상품 DOM 생성이 원인이므로 가상화나 페이지네이션을 제안했다.
- [ ] Profiler Flamegraph에서 `actualDuration`이 높은 컴포넌트를 찾아 병목 지점을 특정할 수 있다고 설명했다.

### 실습 B: Server Components + Streaming SSR 아키텍처 설계

#### 시나리오

실습 A의 `ProductPage`를 Next.js App Router 기반으로 개선하려 합니다.
상품 데이터는 DB에서, 리뷰 데이터는 외부 API에서 가져오며, 리뷰 로딩이 특히 느립니다.

#### 질문

1. Suspense를 적용해 `ProductList`와 `ReviewSection`을 어떤 구조로 분리할 수 있는가?
2. Server Components를 도입하면 어떤 부분이 서버로 이동할 수 있는가?
3. Streaming SSR을 적용하면 사용자 체감이 어떻게 달라지는가?

#### 답안 체크리스트

- [ ] Suspense로 ProductList와 ReviewSection을 독립 boundary로 분리해 점진적 로딩을 구현할 수 있다고 설명했다.
- [ ] Server Components로 데이터 의존 컴포넌트를 서버에서 렌더링해 번들 크기와 초기 로딩을 줄일 수 있다고 설명했다.
- [ ] Streaming SSR로 느린 ReviewSection이 로딩되기 전에 ProductList를 먼저 보여줄 수 있다고 설명했다.

## 오해 바로잡기

1. **"Profiler에서 느린 컴포넌트를 찾으면 바로 React.memo를 적용하면 된다"**
   - 항상 그렇지 않다. Profiler가 보여주는 시간은 "이 컴포넌트가 렌더 대상에 포함된 시간"이지, "이 컴포넌트가 불필요하게 렌더되었다"는 뜻은 아니다. props/state가 실제로 변경되었다면 렌더는 정당하며, 계산 자체를 최적화해야 한다.

2. **"Server Components를 쓰면 Fiber가 필요 없다"**
   - 아니다. Server Components의 결과도 클라이언트에서 Fiber 트리에 통합된다. Fiber 아키텍처는 서버/클라이언트 양쪽의 기반 구조다.

3. **"Streaming SSR은 모든 페이지에 적용하면 좋다"**
   - 반드시 그렇지 않다. 작은 페이지에서는 전체를 한 번에 보내는 것이 오히려 빠를 수 있다. Streaming은 데이터 소스 간 응답 시간 차이가 큰 페이지에서 효과가 크다.
