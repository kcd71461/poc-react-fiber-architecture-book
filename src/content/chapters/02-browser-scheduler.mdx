# 02. 브라우저와 시간을 나누는 Scheduler

## 비유 소개

오케스트라 지휘자는 한 파트만 계속 연주시키지 않습니다.
현악기, 목관, 타악기가 들어와야 할 타이밍을 나눠서 전체 곡이 끊기지 않게 만듭니다.

Scheduler도 같은 역할을 합니다.
React 작업만 몰아서 처리하면 입력/스크롤/페인트가 밀리고, 반대로 너무 자주 멈추면 작업 완료가 늦어집니다.
이제 이 비유를 실제 브라우저 실행 문제로 옮겨보겠습니다.

## 문제 정의

메인 스레드는 React 전용 공간이 아닙니다.

- 사용자 입력 이벤트 처리
- 스타일 계산, 레이아웃, 페인트
- 애니메이션 프레임 처리
- React 외 라이브러리의 스크립트 실행

즉, React는 메인 스레드를 "독점"하는 대신
<InlineAnnotation term="frame budget">
  브라우저가 한 프레임 안에서 처리해야 하는 시간 한도입니다.
  60Hz 환경에서는 약 16.7ms가 기준이며, 이 한도를 반복적으로 넘기면 끊김(jank)과
  입력 지연이 커집니다.
</InlineAnnotation>
안에서 시간을 나눠 써야 합니다.

## 해결 방법

Scheduler의 핵심은 "작업을 작은 단위로 나누고, 필요 시 제어권을 양보한다"입니다.

1. 우선순위가 높은 작업부터 선택한다.
2. 실행 시작 시각을 기록하고(`performance.now()`), 남은 시간을 계산한다.
3. 예산을 소진하면 `shouldYieldToHost`를 `true`로 보고 루프를 멈춘다.
4. `MessageChannel`로 다음 host callback을 빠르게 예약해 이어서 실행한다.

`MessageChannel`이 자주 선택되는 이유는 "다음 턴 예약 지연을 비교적 짧고 일관되게 유지"하기 쉽기 때문입니다.
`setTimeout(fn, 0)`은 환경에 따라 최소 지연 클램핑(대표적으로 4ms 이상)이 걸릴 수 있고,
`requestIdleCallback`은 유휴 시간이 없으면 callback이 충분히 늦게 실행되거나 건너뛰어질 수 있습니다.

| API | 예약 특성 | Scheduler 기본 루프 적합성 | 주의점 |
| --- | --- | --- | --- |
| `setTimeout(0)` | 매크로태스크로 다음 턴 예약 | 낮음 | 클램핑/탭 상태에 따라 지연 편차가 큼 |
| `requestIdleCallback` | 유휴 구간에만 실행 | 낮음 | 유휴 시간이 부족하면 응답 시점 예측이 어려움 |
| `MessageChannel` | 짧은 지연으로 다음 콜백 예약 | 높음 | 직접 큐/중복 예약 제어가 필요 |

## 브라우저 API 역할 분해

코드에 등장할 브라우저 API를 역할별로 먼저 정리하면 아래와 같습니다.

| API | Scheduler에서의 역할 | 오해하기 쉬운 포인트 |
| --- | --- | --- |
| [`performance.now()`](https://developer.mozilla.org/en-US/docs/Web/API/Performance/now) | 현재 slice 경과 시간을 측정 | "시간 측정만 빠르면 성능이 좋아진다"가 아님 |
| [`MessageChannel`](https://developer.mozilla.org/en-US/docs/Web/API/MessageChannel) | 다음 host callback을 짧은 지연으로 예약 | 렌더링 자체를 빠르게 만드는 API는 아님 |
| [`requestAnimationFrame`](https://developer.mozilla.org/en-US/docs/Web/API/Window/requestAnimationFrame) | 프레임 경계에서 시각적 업데이트 타이밍을 맞춤 | Scheduler 루프 자체를 대체하지 않음 |
| [`requestIdleCallback`](https://developer.mozilla.org/en-US/docs/Web/API/Window/requestIdleCallback) | 저우선 후처리 작업에 보조적으로 활용 가능 | React Scheduler의 기본 work loop 타이머는 아님 |
| `setTimeout(0)` | fallback 예약 수단으로 사용 가능 | `MessageChannel`과 동일한 타이밍 특성을 보장하지 않음 |

이제 이 역할 분해를 기준으로, Scheduler 루프 의사 코드를 살펴보겠습니다.

## 기술적 구현 (TypeScript 의사 코드)

아래 코드는 실제 React 내부 구현을 설명용으로 단순화한 의사 코드(pseudo-code)입니다.
위 전략이 코드에서 어떻게 구현되는지, 핵심 루프만 보면 아래와 같습니다.
Scheduler는 작업의 우선순위를 구분합니다. React 내부에서는 이 우선순위를
**Lane**이라는 모델로 표현하며, 06장에서 상세히 다룹니다.
여기서는 "높음 / 보통 / 낮음" 정도의 구분만 이해하면 충분합니다.

```ts
type Priority = "high" | "normal" | "low";

interface Task {
  priority: Priority;
  expirationTime: number;
  callback: () => Task["callback"] | null;
}

const taskQueue: Task[] = []; // 실제 React는 min-heap 기반 우선순위 큐를 사용
const channel = new MessageChannel();

let isHostCallbackScheduled = false;
let sliceStartTime = 0;
const sliceBudgetMs = 5; // 60fps(약 16.7ms) 프레임 예산 안에서 브라우저에 양보할 시간을 남기기 위한 예시 값

channel.port1.onmessage = performWorkUntilDeadline;

function performWorkUntilDeadline(): void {
  isHostCallbackScheduled = false;
  sliceStartTime = performance.now();

  let currentTask = peekHighestPriorityTask(taskQueue); // 우선순위가 가장 높은 작업 조회
  while (currentTask !== null) {
    if (shouldYieldToHost()) {
      requestHostCallback();
      return;
    }

    const continuation = currentTask.callback();
    if (continuation === null) {
      popTask(taskQueue); // 완료된 작업 제거
    } else {
      currentTask.callback = continuation;
    }

    currentTask = peekHighestPriorityTask(taskQueue);
  }
}

function shouldYieldToHost(): boolean {
  return performance.now() - sliceStartTime >= sliceBudgetMs;
}

function requestHostCallback(): void {
  if (isHostCallbackScheduled) return;
  isHostCallbackScheduled = true;
  channel.port2.postMessage(null);
}
```

코드 해석의 핵심은 두 가지입니다.

- `shouldYieldToHost`가 `true`가 되는 순간 현재 slice를 끝낸다.
- 끝낸 후에는 `MessageChannel`로 다음 턴을 예약해 "중단 후 재개"를 구현한다.
- 여기서 `requestHostCallback`은 "다음 틱 예약(schedule next tick)" 역할을 담당하는 함수다.
- `peekHighestPriorityTask`/`popTask`는 우선순위 큐(min-heap)에서 조회/제거를 담당한다.

### 시나리오

```tsx
function Dashboard({ rows }: { rows: Row[] }) {
  const [query, setQuery] = useState("");
  const [chartData, setChartData] = useState<Row[]>(rows);

  function handleChange(next: string) {
    setQuery(next); // 긴급 업데이트

    const recalculated = recomputeHeavyChart(rows, next); // 무거운 연산
    setChartData(recalculated); // 긴급도 낮은 업데이트
  }

  return (
    <>
      <input value={query} onChange={(e) => handleChange(e.target.value)} />
      <Chart data={chartData} />
    </>
  );
}
```

## 오해 바로잡기

1. **"`MessageChannel`을 쓰면 계산 자체가 빨라진다"**
   - 아니다. 핵심은 속도 증가가 아니라 실행 타이밍 조율이다.

2. **"`requestAnimationFrame`만 있으면 Scheduler가 필요 없다"**
   - 아니다. `rAF`는 프레임 경계를 알려주고, Scheduler는 작업 우선순위와 중단 시점을 결정한다.
   - React 내부에서 `rAF`는 Scheduling 루프가 아닌 프레임 동기화 보조 목적으로만 사용된다.

3. **"`requestIdleCallback`을 쓰면 되지 않나?"**
   - React는 초기에 `requestIdleCallback`을 실험적으로 사용했으나, 브라우저 호환성 문제(Safari 미지원)와 유휴 시간 콜백의 실행 타이밍이 일관되지 않아 `MessageChannel` 기반으로 전환했다. `rIC`는 "언제 실행될지 예측하기 어렵다"는 점이 핵심 한계이다.

4. **"`setTimeout(0)`은 `MessageChannel`과 같다"**
   - 완전히 같지 않다. 브라우저 클램핑/지연 특성 때문에 callback 타이밍 안정성이 다르게 나타난다.
